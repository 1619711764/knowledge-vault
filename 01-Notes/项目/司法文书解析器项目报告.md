> 版本 v0.1.0 | 2026-05-29 | Python 3.10+

## 一、项目概述

### 目标

将中国法院裁判文书（PDF/DOCX）自动解析为结构化 JSON，提取案号、法院、当事人、日期、法律依据、判决结果等关键字段。

### 应用场景

- 法律科技公司批量处理判决书
- 律师团队建立案件数据库
- 法学研究者做统计分析
- 企业法务做案例检索

### MVP 范围

| 优先级 | 功能 | 状态 |
|--------|------|------|
| P0 | PDF/DOCX 文本提取 | 已实现 |
| P0 | 段落切分 | 已实现 |
| P0 | 正则抽取（案号/日期/当事人/法条） | 已实现 |
| P1 | LLM 兜底（DeepSeek/通义千问） | 已实现 |
| P1 | CLI 命令行工具 | 已实现 |
| P2 | 裁判文书专用 NER 模型 | 后续 |
| P2 | Web API 接口 | 后续 |

## 二、架构设计

### 四层流水线

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  文本提取      │ ──→ │  段落切分      │ ──→ │  信息抽取      │ ──→ │  LLM 兜底     │
│ (PDF/DOCX)   │     │ (正则关键词)   │     │ (正则+jieba) │     │ (DeepSeek)   │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
                                                                        │
                                                                        ▼
                                                               ┌──────────────┐
                                                               │  结构化 JSON  │
                                                               └──────────────┘
```

### 设计原则

- **策略模式**：ABC 抽象基类确保 PDF 库和 LLM 供应商可替换
- **工厂函数**：`make_extractor(path)` 自动根据文件扩展名选择提取器
- **Pydantic v2**：输出 Schema 层强类型约束，边界校验
- **规则优先 + AI 兜底**：正则能覆盖 60-70% → 省成本；AI 处理剩余 → 保质量

## 三、技术栈

| 层 | 技术 | 选型理由 |
|----|------|---------|
| 文本提取 | PyMuPDF + python-docx | PyMuPDF 比 pdfplumber 快 3-10 倍，中文支持好 |
| 段落切分 | re 正则 + 关键词 | 判决书格式固定，"本院认为""判决如下"等锚点准确率高 |
| 信息抽取 | 正则 + jieba | 案号、日期、法条有强规则模式，正则直接提取 |
| LLM 兜底 | DeepSeek / 通义千问 | DeepSeek 便宜（¥0.001/1K tokens），通义千问国内免费额度 |
| 数据模型 | Pydantic v2 | 类型安全 + JSON Schema 自动生成 + 运行时校验 |
| 分词 | jieba | 最成熟的中文分词库，支持自定义词典 |

## 四、数据模型

### 输出 JSON 示例

```json
{
  "case_no": "(2024)京0105民初12345号",
  "court": "北京市朝阳区人民法院",
  "case_type": "民事",
  "doc_type": "判决",
  "case_cause": "买卖合同纠纷",
  "filing_date": null,
  "judgment_date": "2024-03-15",
  "plaintiffs": [
    {"name": "张三", "role": "原告"}
  ],
  "defendants": [
    {"name": "某某科技有限公司", "role": "被告"}
  ],
  "judges": ["审判长 王五", "审判员 李六", "书记员 钱九"],
  "legal_basis": [
    "《中华人民共和国民法典》第五百七十七条",
    "《中华人民共和国民事诉讼法》第二百六十条"
  ],
  "judgment": "被告于本判决生效之日起十日内支付原告货款50万元...",
  "reasoning": "本院认为，原被告之间的买卖合同关系成立并生效..."
}
```

### Pydantic 模型结构

```
JudgmentDocument (聚合根)
├── CaseNumber → 案号解析(year, court_code, procedure, serial_no)
├── Court → 法院信息(name, level)
├── DocumentDate → 日期(filing, judgment, publication)
├── Party (多) → 当事人
│   ├── PartyAddress → 地址
│   ├── LegalEntity → 法人信息
│   └── Agent (多) → 代理人
├── Judge (多) → 审判人员
├── DocumentSection (多) → 文档分节
├── JudgmentResult → 判决结果
└── applicable_laws → 法律依据列表
```

## 五、抽取规则

### 正则模式库

| 字段 | 正则 | 测试用例 |
|------|------|---------|
| 案号 | `[(]\d{4}[)]...\d+号` | (2024)京0105民初12345号 |
| 日期(数字) | `\d{4}年\d{1,2}月\d{1,2}日` | 2024年3月15日 |
| 日期(中文) | 中文数字→int 转换 | 二〇二四年三月十五日 |
| 法条 | `《...》第X条` | 《民法典》第577条 |
| 金额 | `[\d,]+\s*(?:万)?\s*元` | 人民币50万元 |
| 法院 | `XX人民法院` | 北京市朝阳区人民法院 |

### 段落切分锚点

```
"本院查明" / "经审理查明"  → 事实认定段
"本院认为" / "本院审查认为" → 说理段
"判决如下" / "裁定如下"     → 判项段
"审判长" / "审判员"        → 尾部
```
