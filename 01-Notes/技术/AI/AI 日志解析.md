# AI 日志解析

> 用大语言模型（LLM）自动从非结构化日志中提取结构化信息

## 核心思路

运维日志通常是这样的：

```
2026-05-26 14:32:01 ERROR [db-server-03] connection timeout: 10.0.1.5:3306
```

用 AI 提取其中所有关键信息，输出 JSON 格式 — 一次 prompt 搞定。

## Prompt 设计模板

```
从以下日志中提取时间、级别、服务器名、错误类型、IP和端口，用JSON格式返回。

日志：
<这里粘贴日志>

输出格式：
{"时间": "...", "级别": "...", "服务器": "...", "错误类型": "...", "IP": "...", "端口": "..."}
```

## 不用 AI 的情况

以下情况直接用正则就够了，不需要 AI：
- 日志格式固定、规则明确
- 数据量大（每天百万行），AI 成本太高
- 实时性要求高（毫秒级）

| 场景 | 方案 |
|------|------|
| 格式固定 | [[正则表达式实战]] — Python `re` 模块 |
| 格式多变 | AI Prompt + JSON 输出 |
| 海量日志 | grep 预处理 → 正则批量解析 → ES 聚合 |
| 少量复杂日志 | 直接丢给 AI |

## 实际工作流

```
                     ┌──────────┐
   原始日志 ──────→  │ grep/正则  │ ──────→ 结构化字段
                     │ 批量预处理  │
                     └──────────┘
                           │
                     ┌──────▼─────┐
                     │ AI 处理复杂 │  ← 格式不固定的丢给 AI
                     │ 或异常的日志│
                     └────────────┘
                           │
                     ┌──────▼─────┐
                     │ ES / 数据库 │  ← 结构化后入库查询
                     └────────────┘
```

## Prompt 工程要点

1. **明确输出格式** — 告诉 AI 要用 JSON，字段有哪些
2. **给示例** — Few-shot 示例大大提高准确率
3. **约束类型** — "端口是数字"、"IP 是 x.x.x.x 格式"
4. **批量处理** — 一次给多条日志，让 AI 返回 JSON 数组

```
以下是多条日志，逐条提取信息，返回 JSON 数组：

[日志 1]
[日志 2]
[日志 3]

输出格式：[{"时间":..., "级别":...}, ...]
```
